# 技能系统搭建指南

> 供其他 Agent 使用的技能系统设计规范与操作手册。
> 本 skill 不直接创建技能，而是指导 Agent 如何正确设计和搭建技能体系。

## 适用场景

- 其他 Agent 需要从零搭建一套技能系统
- 需要创建一个全新的 skill 并注册到系统中
- 需要修改/优化现有 skill 的结构，统一格式
- 需要对多个 skill 进行批量标准化
- 需要在创建/修改技能后执行内容变更同步

## 技能系统核心概念

技能系统由三个核心要素构成：

```
技能文件（SKILL.md）        →  技能的定义和内容
技能存储目录                →  技能的物理组织形式
技能注册机制                →  技能被发现和调用的方式
```

这三个要素应遵循统一的命名规范、目录结构、内容模板，否则技能系统将快速演变为不可维护的混乱集合。

## 目录结构设计

### 通用目录布局

```
<Skills Root>/
├── <skill-name>/           # 每个 skill 一个独立文件夹
│   ├── SKILL.md            # 必选：技能定义文件（YAML frontmatter + 正文）
│   ├── scripts/            # 可选：Python/bash 可执行脚本
│   ├── references/         # 可选：参考文档、数据表、配置模板
│   ├── agents/             # 可选：子 Agent 配置（多 Agent 场景）
│   └── assets/             # 可选：图片、模板等静态资源
├── shared/                 # 可选：跨 skill 共享库（不自成 skill）
└── <other-top-level-dirs>/ # 根据平台需求定制的其他目录
```

### 设计原则

1. **单文件夹原则**：每个 skill 一个独立文件夹，文件夹名即为 skill 标识名
2. **SKILL.md 为核心**：这是技能的唯一入口文件，其他文件都是辅助
3. **资源就近存放**：技能相关的脚本、文档、模板放在技能自己的目录下，不散落全局
4. **共享库隔离**：跨 skill 共享代码放在 `shared/` 之类专用目录，不重复复制也不混淆

## SKILL.md 标准模板

### YAML Frontmatter（必须）

```yaml
---
name: <skill-name>           # 必填：技能标识名，与文件夹名一致
description: |               # 必填：技能说明 + 触发词
  一句话描述技能用途。
  触发词：关键词1、关键词2...
version: "1.0.0"             # 推荐：语义版本号
---
```

`name` 是技能系统的**唯一标识**，用于注册、引用、触发匹配。`description` 决定触发精度，应覆盖用户可能使用的多种表述方式（同义表达、常见口语、中英文变体）。

### 正文结构

#### 工具执行型（有脚本、命令行工具）

这类 skill 封装了可执行的命令行工具，推荐按以下 8 节模板组织：

```
# <Title>

## 适用场景
- 列出何时应该激活此 skill
- 用户口中什么样的表述会触发

## 必要输入
- 参数表：参数名、说明、数据类型、默认值

## 依赖
- 外部工具、运行时环境、Python 包等

## 执行步骤
### Step 0 环境检查 → Step 1~N 逐步操作

## 错误处理 / 失败分流
| 错误情景 | 原因分析 | 解决方案 |

## 输出约定
- 操作完成后向用户汇报的内容格式和示例

## 边界定义
### 不该激活 — 什么情况下应改用其他 skill
### 不该做 — 明确禁止的操作行为
### 不该碰 — 不越界触碰的文件/硬件/系统配置

## 交接关系
- 上游 skill（调用本 skill 之前应先执行的）
- 下游 skill（本 skill 执行后应继续的）
- 互补 skill（功能有重叠但侧重点不同）
```

#### 知识参考型（无脚本，纯技术文档）

这类 skill 是技术知识库，不封装可执行脚本。结构可根据内容灵活调整，但至少应包含：

```
# <Title>

## 适用场景
## 必要输入 / 使用前提
## 边界定义
## 交接关系
```

其他章节（技术详解、使用指南、常见陷阱、配置参数、平台差异）按需添加，不强制模板化。

## 创建流程

### 阶段一：规划
1. 确定 skill 类型：工具执行型 or 知识参考型
2. 确定 skill 名称（遵守命名规范）
3. 确定分类归属（属于技能体系中的哪个领域）

### 阶段二：创建
1. 在技能目录下创建以 skill 名称命名的文件夹
2. 写入 `SKILL.md`，包含完整的 YAML frontmatter
3. 按类型选择合适的正文模板进行填充
4. 如有脚本/文档，放入对应的 `scripts/`、`references/` 子目录

### 阶段三：注册
将 skill 注册到平台的技能发现机制中，注册内容包括：
- 技能名称（与文件夹名一致）
- 描述信息（取自 SKILL.md 的 description 字段）
- 启用状态

### 阶段四：验证
1. 确认 skill 可在会话中被正确触发
2. 确认 description 中的触发词覆盖了常见表达
3. 确认边界定义中的"不该激活"场景不会错误触发

### 阶段五：同步
创建/修改 skill 后，同步更新技能体系中的关联文档：
```
1. 主灵魂文件（SOUL.md / 等效文件）— 技能清单和映射表
2. 知识/事实文件（FACT.md / 等效文件）— 系统变更记录
3. 技能地图（如有）— 分类表和推荐矩阵
4. 运行时配置文件（如有）— 全局技能列表
```

## 命名规范

### 通用规则

| 规则 | 示例 |
|------|------|
| 小写英文字母 + 连字符 | `my-skill`、`some-module` |
| 动词在前（工具执行型） | `build-project`、`flash-device` |
| 名词在前（知识参考型） | `bus-protocol`、`core-architecture` |
| 统一后缀表达同类 skill | `-module`（知识型）、`-debug`（调试型）、`-tool`（工具型） |
| 避免中文名 | 不取 `我的技能` |
| 避免驼峰和下划线 | 不取 `MySkill`、`my_skill` |
| 避免版本号 | 不取 `my-skill-v2` |

### 分类前缀/后缀

合理的命名体系能让技能按名称排序时自动归类。例如：
- `*-framework` → 框架型技能
- `*-module` → 知识模块
- `*-debug` / `*-tool` → 调试/工具
- `build-*`/`flash-*` → 构建/烧录流水线

命名体系应根据具体领域设计，关键是一致性。

## Description 编写规范

description 字段决定技能的触发匹配精度，遵循以下原则：

```
结构：
  一句话说明功能 + 换行 + 覆盖性触发词列表

要点：
  - 第一句直截了当：这个技能做什么？
  - 触发词覆盖同义表达（正式用语、口语、简称、常见拼写错误）
  - 中英文双语覆盖（如适用）
  - 长 description 用 YAML 块标量 | 而不是折叠标量 >
  - 不写版本号、依赖信息、示例代码（正文中放）
```

示例（通用框架）：
```yaml
description: |
  分析项目构建产物的大小和资源占用。
  提供优化建议、版本对比、历史趋势追踪。
  触发词：构建分析、产物大小、资源占用、包体积优化、
  bundle analysis、size report、diff build。
```

## 错误处理

| 错误 | 原因 | 解决方案 |
|------|------|---------|
| 技能无法被触发 | name 或 description 字段缺失 | 确认 YAML frontmatter 完整 |
| 触发命中不准确 | description 中触发词过少 | 补充同义表达、常见变体 |
| 文件夹名与 name 不一致 | 手动创建时疏忽 | 确保两者完全一致 |
| 文件编码异常 | 非标准 UTF-8 编码 | 保存为 UTF-8（无 BOM 或带 BOM 均可） |
| 边界条件漏处理 | 未定义"不该激活"场景 | 补充边界定义，列出应使用其他 skill 的场景 |
| 新 skill 与已有 skill 冲突 | 命名重复或功能重叠 | 检查现有技能体系，考虑合并或重命名 |
| 脚本类 skill 缺少依赖 | 未声明运行时依赖 | 在「依赖」章节列出所有外部工具和包 |

## 输出约定

创建/修改 skill 后输出以下信息供检查：
- skill 名称和目录路径
- YAML frontmatter 完整性（name / description / version 是否齐全）
- 正文章节覆盖情况（对照模板检查）
- 变更同步检查结果（关联文档是否已更新）

## 边界定义

### 不该激活
- 用户需要的是一个具体的**功能实现**（如"写一个函数"、"生成一段代码"）而非技能系统设计
- 用户需要的是对**现有单一技能**的优化改进（应直接编辑该技能，而非启动整个搭建流程）
- 用户只想要了解技能系统有哪些可用技能（应使用技能地图/导航类 skill）
- 用户的诉求是纯概念性讨论，没有实际创建/修改需求

### 不该做
- **禁止**创建多层嵌套的技能目录（不超过 2 级）
- **禁止**省略 YAML frontmatter 中的 `name:` 或 `description:` 字段
- **禁止**在 SKILL.md 中使用占位符（"TODO"、"待定"、"可能"、"请填写"）
- **禁止**创建文件夹名与 `name:` 不一致的 skill
- **禁止**多个 skill 共用同一个文件夹
- **禁止**在 SKILL.md 中写入安全凭证、API Key、私钥等敏感信息

### 不该碰
- **不触碰**共享库目录（其他技能依赖的公用代码）
- **不触碰**非本体系的技能（不代表其他技能系统做优化）
- **不触碰**系统级配置文件（不属于技能体系管理的文件）
- **不触碰**其他 Agent 正在使用的技能（除非有明确的跨 Agent 同步机制）

## 交接关系

- 上游：**技能规划阶段**（确定要创建什么技能、解决什么问题）
- 互补：**技能地图/导航 skill**（导航查看已有技能，本 skill 指导创建新技能）
- 下游：**技能质量评估**（新建 skill 后可对其做结构完整性和触发精度评估）
- 创建工具执行型 skill → 参考系统中已有工具型 skill 的模板
- 创建知识参考型 skill → 参考系统中已有知识型 skill 的模板
- 创建完成 → 更新技能地图和技能体系统计

## 通用设计原则

### 一个 skill 只做一件事
如果一份 SKILL.md 描述了多个不相关的功能，应拆分为多个 skill。

### 描述即入口
用户和系统通过 description 找到技能，description 的质量直接决定技能的可用性。

### 边界即契约
"不该做"和"不该碰"定义了技能的职责边界，是技能体系稳定运行的保障。

### 同步不是可选项
创建或修改技能后不更新关联文档，技能体系会在几次迭代后快速失序。
