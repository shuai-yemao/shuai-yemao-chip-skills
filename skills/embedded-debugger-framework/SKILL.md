# 嵌入式系统故障诊断框架
> 主题 Skill | 综合 Arnold Berger、Jack Ganssle、Embedded Artistry 等嵌入式调试领域的共识方法论

## 核心原则

**调试不是试错，是一层层排除的推理过程。** 嵌入式系统的故障诊断有一条铁律：永远从最底层开始，逐层向上验证。跳过任何一层都是赌博。

## 框架概览：五层诊断模型
嵌入式系统故障必定落在这五层之一。你的工作是逐层排除，而非猜测。

| 层 | 检查对象 | 工具 | 典型根因 |
|---|---------|------|---------|
| **L0 供电** | VDD/VDDA/VBAT 电压、纹波、电流 | 万用表、示波器 | LDO 过热保护、电容失效、焊接虚焊 |
| **L1 物理连接** | 引脚焊接、线序、电平匹配 | 万用表蜂鸣档、示波器 | TX->RX 反接、3.3V->5V 不匹配、冷焊 |
| **L2 波形/时序** | 信号质量、波特率、时序参数 | 逻辑分析仪、示波器 | 波特率偏差、时钟不准、SPI 模式错 |
| **L3 寄存器/配置** | 外设寄存器状态、时钟使能、AFIO | 调试器、printf | IDLE 预设、时钟未使能、DMA 配置错 |
| **L4 软件/状态机** | ISR 设计、任务同步、缓冲区管理 | 调试器、日志 | ISR 阻塞调用、竞态条件、栈溢出 |

**方向选择**：有新板/新硬件 -> 自底向上（L0->L4）；已工作过的板突然出问题 -> 从现象层往两边展开

## 诊断工作流
### 阶段 1：复现与缩小范围

```
1. 精确复现条件：什么操作序列？什么环境？发生频率？
2. 最小化系统：去掉不相关模块（断电无关外设、断开无关线缆）
3. 如果去掉了某个模块 bug 消失 -> 该模块是"冒烟点"
4. 构造最小测试用例：不跑完整应用，仅跑出问题的那条路径
```

**关键原则**：不要边猜边改。先做到「每次都能复现」，再开始诊断。

### 阶段 2：假设驱动诊断
```
1. 列出所有可能导致该现象的假设（<=5 个）
2. 按"最容易验证"排序，而不是按"最可能"
3. 每个假设设计一个排他性实验（能区分 A 和 not-A）
4. 一次只改一个变量，改了就跑实验
```

**为什么最容易验证的优先**：用示波器量电压 30 秒比改代码重编译烧录 5 分钟快 10 倍。

### 阶段 3：二分法隔离

故障定位的最快方法不是线性排查，是二分法：
```
全系统正常 -> [不] -> 断开后半段 -> 前半段正常
  -> [是] -> 问题在后半段 -> 后半段再分两半 -> ...
  -> [否] -> 问题在前半段 -> 前半段再分两半 -> ...
```

在嵌入式系统的具体应用：
- 软件层：在数据流中间插打印 -> 确定数据在哪一步变错
- 硬件层：飞线跳过中间电路 -> 确定哪一级引入异常

## 核心诊断模式

### 模式 1：UART/DMA 变长接收三板斧
这是 STM32 上最容易踩的坑。`HAL_UARTEx_ReceiveToIdle_DMA` 有三道坎：
```
[1] 清 IDLE 标志：__HAL_UART_CLEAR_IDLEFLAG(huart)
    根因：USART_SR.IDLE 硬件复位后默认为 1，第一次启动 DMA 立即触发 Size=0 的回调
[2] 读 SR 寄存器诊断：检查 FE(帧错误)/ORE(溢出)/NE(噪声) 标志
    - FE=1 + 数据全 0x00 -> RX 线被硬件拉低/断路/break 条件
    - ORE=1 -> 接收缓冲区溢出，DMA 来不及取数据
    - NE=1 -> 线路噪声，检查接地和屏蔽

[3] ISR 只置标记不阻塞：回调里绝对不调用 HAL_UART_Transmit/HAL_Delay 等阻塞函数
    根因：阻塞 HAL 函数检查 huart->gState，在 ISR 上下文中 gState 已被锁定
    结果：gState 永久卡在 BUSY，后续所有 UART 操作全部失效
```

### 模式 2：HardFault 诊断

Cortex-M 的 HardFault 是可诊断的——不要用 `while(1)` 敷衍：
```
1. 读 CFSR (Configurable Fault Status Register) @ 0xE000ED28
   - UFSR(位 15-8): 未定义指令、除零、未对齐访问
   - BFSR(位 7-0): 总线错误（访问非法地址）
   - MMFSR(位 7-0): 内存管理错误

2. 从栈帧恢复现场：
   - 读 MSP/PSP 找到栈指针
   - 解析栈帧：R0-R3, R12, LR, PC, xPSR
   - PC 就是触发异常的指令地址 -> .map 文件反查函数名
3. 常见根因：
   - 栈溢出（覆盖了栈底，LR 被破坏）-> 加大栈 / 检查 canary
   - 空指针解引用 -> 检查外设是否已初始化
   - 外设时钟未使能就访问寄存器 -> BusFault
   - 除零（UsageFault）-> 检查除数变量路径
```

### 模式 3：栈溢出检测
最隐蔽的 bug 之一：「一切看起来都正常，但偶尔莫名其妙地跑飞或复位」。
```
快速检测：
1. 把所有任务栈大小 x2 -> 问题消失 = 栈溢出的强证据
2. 在栈末端写 canary 值（如 0xBEEF），定期检查是否被覆盖
3. FreeRTOS 开启 configCHECK_FOR_STACK_OVERFLOW = 2
4. MPU 方案：在栈下方设保护区 -> 溢出触发 MemManage Fault
```

### 模式 4：Cache 一致性
Cortex-M7 的 DCache + DMA 是高频陷阱：
```
现象：DMA 搬完数据，CPU 读到的却是旧值（或反过来）
根因：CPU 读的是 Cache 中的旧数据，DMA 写的是物理 RAM

解法：
- 快速验证：关闭 DCache——问题消失即确认
- 正式修复：DMA 接收 -> SCB_CleanInvalidateDCache(buf, len)
-           DMA 发送 -> SCB_CleanDCache(buf, len)
- 终极方案：DMA 缓冲区放在 non-cacheable 区域（MPU 配置）
```

### 模式 5：TX 返回 OK 但实际输出异常
`HAL_UART_Transmit` 返回 `HAL_OK` 不代表引脚上有正确的数据：
```
验证方法：
1. 逻辑分析仪/示波器直接量 TX 引脚——看波形是否存在、电平是否正确
2. 上位机回环测试：TX 和 RX 短路 -> 看收到的是否和发出的一致
3. 检查 AFIO 映射：TX 引脚是否正确配置为 AF_PP
4. 检查 GPIO 时钟和 USART 时钟是否都已使能
```

### 模式 6：J-Link/ST-Link 暂停态寄存器读数为假象
调试器暂停核心时，外设寄存器的读数不是运行时的真实状态——这是最多人踩的陷阱之一：
```
现象：通过调试器读取 RCC_APB2ENR 发现 = 0（时钟未使能？）
      GPIOA_CRL = 0x44444444（浮空输入？）
      SysTick 相关寄存器 = 0（SysTick 没工作？）
      但程序明明在上电后正常跑了很久
根因：调试器（J-Link/ST-Link）暂停(halt)核心时，部分外设总线时钟被调试接口关闭
      或寄存器值在 halt 状态下回读异常，导致读到的全是复位值或默认值
诊断方法：
1. 信任运行时证据 > 暂停态寄存器值
   - LED 在正常闪烁 -> GPIO 一定配置正确
   - 串口能输出数据 -> UART 时钟已使能
   - SysTick 中断触发 -> SysTick 已配置
2. 用 GPIO 翻转法替代寄存器读取来验证运行时状态
3. 在代码中插入寄存器回读+打印（运行时读取），而非暂停时读取
4. 用 DWT->CYCCNT 或定时器捕获功能测量运行时时序
```

### 模式 7：单总线协议 (DHT22/DHT11 等) 调试

单总线协议（open-drain + 外部上拉）的调试难点在于时序敏感且无时钟线：
```
现象：传感器有时能读到数据有时超时，或读到的数据值与实际差异巨大

排查顺序（由外到内）：
1. L0 供电：
   - DHT22 VCC 必须接 5V，3.3V 超出规格下限
   - 电源纹波 < 100mV，供电电流 > 10mA 每个传感器
2. L1 物理连接：
   - DATA 线必须有外部上拉，4.7k 推荐，STM32 内部 ~40k 上拉不足
   - 导线长度 < 20m（长线增加寄生电容，影响上升沿时序）
   - VCC/GND/DATA 三线连接正确（DHT22 引脚间距 2.54mm）
3. L2 波形/时序：
   - 示波器量 DATA 引脚：复位信号（主机拉低 >= 18ms）是否存在？
   - 应答信号：传感器是否拉低 80us + 拉高 80us？
   - 数据位 "0"：高电平 ~26-28us，数据位 "1"：高电平 ~70us
   - 采样点应在高电平开始后 30us（DHT22）或 40us（DHT11）
4. L3 寄存器/配置：
   - GPIO 模式切换：输出开漏（复位阶段）-> 输入上拉（读取阶段）
   - STM32F1 HAL_GPIO_Init(GPIO_PULLUP) 可能不生效 -> 直接写 CRL
   - 微秒延时精度：禁止用 HAL_Delay(us)（精度 1ms）-> 用 DWT 周期计数器
5. L4 软件/状态机：
   - FreeRTOS 下：40 位数据读取必须用 taskENTER_CRITICAL() 保护
   - 读取间隔 >= 1 秒（DHT22 采样频率上限 1Hz）
   - 连续三次失败才报错，单次失败允许重试
```

### 模式 8：超声波 (HC-SR04) 不工作诊断
HC-SR04 是最容易因硬件接线导致"代码正确但无数据"的外设：
```
现象：读距离始终返回 0（Echo 超时）
排查顺序（由外到内）：
1. L0 供电：HC-SR04 Vcc 必须接 5V（3.3V 不工作）
2. L1 物理连接：
   - Trig 到目标引脚（万用表蜂鸣档确认导通）
   - Echo 到目标引脚（HC-SR04 Echo 输出 5V，建议 2k+1k 分压至 3.3V）
   - GND 共地
3. L2 波形/时序：示波器量 Trig 脚 -> 是否有 >=10us 的高电平脉冲？
   - 无脉冲 = 软件 Trig 没发出（检查 GPIO 配置和时钟）
   - 有脉冲但 Echo 无响应 = HC-SR04 损坏或供电不足
   - 有脉冲 Echo 有响应但代码读不到 = 电平匹配问题（5V->3.3V）
4. L3 寄存器/配置：
   - Trig 引脚：GPIO_Mode_Out_PP 确认（可示波器量到电平跳变）
   - Echo 引脚：GPIO_Mode_IN_FLOATING 或 IPU 确认
   - TIM2: SystemCoreClock / 1MHz - 1 预分频正确？
     SystemCoreClock 是否通过 SystemCoreClockUpdate() 正确初始化
5. L4 软件/状态机：临时加一段独立测试代码剥离业务逻辑
   -> while(1) 中只跑 Ultrasonic_ReadDistanceCm() + 翻转LED
   -> LED 规律翻转 = 传感器硬件正常，问题在任务调度或数据链路中
常见陷阱：
- HC-SR04 最小探测距离 ~2cm，物体太近测不到返回 0
- 测量周期 < 60ms 会触发模块内部保护（连续测量间隔 >= 60ms）
- Echo 下拉电阻不可用太大（建议 10k max），否则影响回波电平识别
```

## 调试工具分层策略

| 场景 | 首选工具 | 次选 |
|------|---------|------|
| 系统彻底不启动 | LED/GPIO 翻转 | 调试器 |
| 运行时异常但不崩溃 | 结构化日志 | assert |
| 通信问题 | 逻辑分析仪 | 示波器 |
| 时序问题 | GPIO 翻转 + 示波器 | DWT CYCCNT |
| HardFault/崩溃 | 调试器 + CFSR 解析 | coredump |
| 栈溢出 | canary 值 | MPU |

**GPIO 翻转调试法**：在关键代码路径 toggle 一个 GPIO，用示波器量。可以精确测量 ISR 进入/退出时间、任务切换延迟、DMA 完成时序。

**DAC 调试法**：用一个 DAC 通道输出不同电压来表示内部状态码，示波器直接读取。适合无串口/无调试器的场景。

## 反模式（绝对不做的事）
1. **盲改**：不改代码就不知道是不是修好了 -> 你根本没理解根因
2. **一次改多个东西**：改了三行代码，重新编译烧录，bug 消失了 -> 你不知道是哪一行修的，也不知道 root cause
3. **用 `while(1)` 吞掉 HardFault**：相当于把"发动机故障灯"剪掉
4. **在 ISR 里加 `HAL_Delay` 等阻塞调用**：永远不要在中断上下文做任何可能阻塞的操作
5. **怀疑编译器**：99.9% 的情况不是编译器 bug，是你自己的代码有问题
6. **跳过供电和物理层**：很多"软件 bug"其实是供电纹波/虚焊/线序反了

## 诚实边界

- 本框架覆盖通用嵌入式系统（ARM Cortex-M / RISC-V / ESP32 等），不覆盖 Linux 内核级调试
- 射频相关故障（BLE/WiFi 丢包、天线匹配）需要频谱仪等专用工具，本框架不深入
- 车规功能安全的诊断要求（ISO 26262 FMEDA）超出本框架范围
- 涉及具体 MCU 的勘误表问题（silicon bug）不在覆盖范围——先查勘误表
