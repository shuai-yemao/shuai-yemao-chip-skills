# 真伪验证引擎

> 搜索结果的每条技术断言都需经过多维交叉验证，不可盲目采信。

## 验证脚本

```bash
# 脚本位置（共享 knowledge-base-search 的脚本目录）
python ../knowledge-base-search/scripts/verify_claims.py --claim "STM32F103 USART1 TX引脚为PA9"

# 从 kb_search JSON 输出管线验证
python ../knowledge-base-search/scripts/kb_search.py "DMA 双缓冲" --json | python ../knowledge-base-search/scripts/verify_claims.py --kb-stdin
```

## 三维评分模型

| 维度 | 权重 | 判定方法 |
|------|------|---------|
| **来源权威性** (40%) | RM/DS=1.0, AN/ES=0.9, HAL源码=0.85, GitHub=0.5, 论坛FAE=0.4, 论坛普通=0.3, B站知名UP主=0.3, B站=0.2, 博客=0.25 | 匹配 source 关键词 |
| **内容安全检查** (30%) | 检测 "仅供参考/未验证/TODO/可能/猜测" 等危险信号 | 正则匹配扣分 |
| **断言交叉验证** (30%) | 提取寄存器值/API签名/时序参数，与本地KB多源比对 | 2+ 独立源一致→VERIFIED |

## 技术一致性自动检查

嵌入式特化规则，自动检测常见错误模式：

| 规则 | 检测内容 | 标签 |
|------|---------|------|
| F103 AFIO 时钟 | STM32F103 + USART1/SPI1 但未提 AFIO 时钟使能 | `[!]` |
| 中断回调阻塞 | IRQHandler/Callback 中调用 `HAL_Delay()` | `[X]` |
| DMA DBM 互斥 | 双缓冲模式 + Memory-to-Memory 同时出现 | `[!]` |
| 译文风险 | "仅供参考""翻译""译文"标记 | `[!]` |

## 判定阈值

| 分数 | 标签 | 含义 |
|------|------|------|
| ≥0.70 | `[VERIFIED]` | 至少2个权威源一致确认，可直接采信 |
| ≥0.45 | `[?] UNCERTAIN` | 仅单源或无权威源可验证，建议交叉比对 |
| ≥0.20 | `[!] DISPUTED` | 权威源间存在矛盾或发现危险信号 |
| <0.20 | `[X] LOW CONFIDENCE` | 被权威源明确否定，不应采信 |

## 使用时机

### 何时激活
- 需要对搜索结果进行可信度评估
- 用户问"这个说法对吗"、"这个寄存器值对不对"
- 多个搜索结果之间不一致时
- 从论坛/博客获取的信息需要验证

### 何时不激活
- 直接来自厂商官网的数据手册内容（官网本身即为最高权威）
- 纯代码逻辑问题（不需要验证）

### 边界
- **不修改**搜索结果原文
- **不写入**知识库（那是 kb-import 的职责）
- **不猜测**权威评分标准外的来源
